# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`drupilot` is **not an application** — it is a **Claude Code plugin** that ports Drupal 9/10 modules and themes to Drupal 11. There is no build step, package manager, or compiled output: the plugin is composed of markdown (commands, agents, skills), JSON (manifests, config, hooks) and a Bash script library. End-user documentation lives in `README.md` (English) / `README_es.md` (Spanish translation). The authoritative product spec is `../PROMPT.md` — its §1 carries verified (June 2026) ecosystem versions and toolchain commands; treat those as facts rather than re-researching.

## Language convention (important — overrides PROMPT.md)

All **functional content** of the plugin is **English only**: comments, log lines, user-facing messages, prompts, reports, frontmatter `description` fields. `PROMPT.md` §0/§7 specifies Spanish output, but that was **explicitly reversed by the user**. The single exception is `README_es.md`, the Spanish translation of `README.md` — if you change `README.md`, mirror it there. Never reintroduce Spanish into any other file. Code identifiers, package names and shell commands stay in their original form.

## Developing / validating the plugin

There is no test runner for the plugin itself. The verification gates are:

```bash
claude plugin validate .            # manifest + command/agent/skill frontmatter + hooks.json
bash -n path/to/script.sh           # syntax-check a shell script (shellcheck is not assumed present)
chmod +x scripts/*/*.sh hooks/scripts/*.sh   # scripts and hook scripts must be executable
```

Smoke-test a script directly (set `CLAUDE_PLUGIN_ROOT` so path/config resolution works outside an installed plugin):

```bash
export CLAUDE_PLUGIN_ROOT="$PWD"
bash scripts/env/preflight.sh --profile all          # human report
bash scripts/env/preflight.sh --profile setup --json # JSON + exit-code gate (0 ok / 2 missing hard req)
bash scripts/env/detect-php.sh --json
```

Most scripts support `-h/--help`, validate their args (clean English error + non-zero exit on misuse), and have a `--json` and/or `--dry-run` mode.

To exercise the plugin end-to-end, install it locally: `/plugin marketplace add .` then `/plugin install drupilot@drupilot`, and start a new session (hooks load at SessionStart).

## Architecture — how the pieces compose

The flow is **command → (gate) → skill + subagent → scripts → templates**, all reading one config and one shell library.

- **`scripts/lib/common.sh` is the contract everything depends on.** Every script sources it (group scripts via `../lib/common.sh`; hook scripts via `../../scripts/lib/common.sh`). It provides logging (`log_info/ok/warn/err/step`, `die`), tool/version detection (`have_cmd`, `tool_version`, `version_ge`, `docker_daemon_up`, `ddev_running`), config access, subject detection (`find_drupal_root`, `subject_type`, `subject_machine_name`, ...), the path/cache helpers (note the split: `project_state_dir` is the HIDDEN per-project machine state under `$HOME` — `assess.json`/`last-test.json`/the lockfile, kept out of the tree so it survives `git clean` and never leaks; `project_artifacts_dir` is the VISIBLE, gitignored `<root>/.drupilot/` for human-facing outputs — `port-report.md`/`viability-report.md`/coverage/the local patch), the **determinism/lockfile helpers** (`deterministic_mode`, `lock_get`/`lock_set`/`lock_set_json`, `lock_resolve`, `lock_show`/`lock_clear`, `drupilot_lock_file`), the **interaction helpers** (`confirm`, `choose_one`, `tty_readable`, `announce_patch`), and the **prefs helpers** (`drupilot_prefs_file`, `prefs_set`, `config_enum`). **Do not reinvent these in individual scripts.**

- **stdout/stderr discipline:** all `log_*` output goes to **stderr**; **stdout is reserved for parseable payloads** (JSON, file lists). Anything a command parses from a script comes from stdout. The analysis scripts (`run-rector.sh`/`run-phpstan.sh`/`run-phpcs.sh`) expose a `--json` mode for a reproducible, machine-readable verdict (native `--error-format=json`/`--report=json`; a `{changed_files,...}` summary for Rector).

- **Config resolution (`config/defaults.json` + `config_get`):** resolution order is **env var > per-project `.drupilot.json` > `config/defaults.json` > caller default** — the env var always wins. The `.drupilot.json` tier lives at the Drupal ROOT and is where in-flow **tabbed choices** are remembered (`prefs_set` writes it; gitignored via `ensure-gitignore.sh` so it never lands in a patch). `config_enum` validates a closed-set key (used non-fatally in `preflight.sh`); a tabbed choice can also be pre-answered with `DRUPILOT_CHOICE_<KEY>`. `DRUPILOT_PHP_TARGET` (default `8.3`) drives all tuning (Rector php sets, PHPStan level, PHPCS, DDEV `php_version`); read it via `resolve_php_target`, never hardcode a version. Versions flagged uncertain (notably PHP 8.5) are gated through `php_target_unconfirmed` and detected at runtime — never assumed.

- **Determinism + lockfile (`DRUPILOT_DETERMINISTIC`, default `true`):** ports are reproducible by default. The resolved Drupal core, dev toolchain, digests commit SHA and DDEV add-ons are frozen in a per-project `drupilot-lock.json` and reused on later runs via the `lock_resolve` *resolve-and-freeze* pattern; `DRUPILOT_DETERMINISTIC=false` resolves fresh and refreshes the lock. **The lock is keyed by the Drupal ROOT** (the only path every context knows), so it lives in a different state dir than `assess.json`/`last-test.json` (keyed by the subject) — readers derive it via `DRUPILOT_PROJECT_DIR=$ROOT drupilot_lock_file` (`assess.json`/`last-test.json` are keyed by the subject instead). `scripts/env/lock-sync.sh` captures/refreshes it (reads `composer.lock` + `ddev add-on list`; called by `ddev-up.sh` and `ddev-add-ons.sh`); `lock_show`/`lock_clear` let a developer inspect or reset it, and `/drupilot-status` prints it. Toolchain constraints in `defaults.json` stay at the verified PROMPT §1 ranges; the lock pins the exact patch. Never invent patch versions.

- **`scripts/env/preflight.sh` is the requirements gate engine**, reused three ways: the `/drupilot-doctor` command, the SessionStart hook, and a per-command gate. It validates only what a profile needs (`analyze` → git+jq+composer/php; `setup`/`test` → Docker+daemon+DDEV; `contribute` → git+SSH/PAT), prints a human table or `--json`, and exits `0` (ready) / `2` (a hard requirement is missing) — `profile=all` always exits 0. Commands call it first and abort with the actionable report and **no side effects** when it fails.

- **Host vs. DDEV execution (`drupal_runner`):** analysis/test scripts `cd` to the Drupal root and prefix toolchain calls with `$(drupal_runner)`, which yields `ddev exec` when the DDEV environment is up and an empty string (host `vendor/bin`) otherwise. Relative paths like `web/modules/custom/foo` resolve identically inside the container and on the host — this is why scripts must cd to the root and use relative subject paths. A **loose** subject (a module/theme not yet under a Drupal root) is never scaffolded on top of: `scripts/env/resolve-workspace.sh` (a pure `--json` resolver, like `core-strategy.sh`) targets a sibling `<name>-d11` test-bed (or `DRUPILOT_WORKSPACE_DIR`) and `scripts/env/place-subject.sh` (idempotent, `--dry-run`, runs after `ddev-up.sh`) moves/symlinks/copies it under `web/{modules,themes,profiles}/custom/<name>` per `DRUPILOT_PLACEMENT` (`move`|`symlink`|`copy`, default `move`); a module already inside a Drupal root keeps its layout (back-compat).

- **Hooks are fail-safe by contract** (`hooks/hooks.json` + `hooks/scripts/*.sh`): they read JSON on stdin, **never use `set -e`**, **never exit non-zero**, guard every optional tool with `|| true`, and act only by printing `{"hookSpecificOutput":{...}}` on stdout (fail-safe default = exit 0 with no JSON). `PreToolUse`/`guard-contrib.sh` returns `permissionDecision: "ask"` for outward-facing git push/MR commands in `semi` mode **and always when `DRUPILOT_AUTONOMOUS=true`** (the backstop enforcing "an autonomous run never pushes on its own", regardless of `DRUPILOT_CONTRIB_MODE`). `PostToolUse`/`post-edit-lint.sh` is **phase-aware** (Phase 1 surfaces compatibility ERRORS only) and controllable via `DRUPILOT_POST_EDIT_LINT` (`autofix`/`report`/`off`), and it reports when phpcbf modified a file; `SessionStart`/`session-detect-env.sh` honors `DRUPILOT_SESSION_CONTEXT` (`on`/`off`).

- **Two-phase porting philosophy:** Phase 1 (`/drupilot-port`, skill `minimal-port`) makes the code D11-compatible with the smallest diff, preserving original behavior; Phase 2 (`/drupilot-refactor`, skill `full-refactor`) is an opt-in "Drupal 11 way" rewrite — now **granular**: a multi-select of modernizations (attributes / DI / strict types / `final` / deprecation-removal) + a PHPStan level, persisted as `DRUPILOT_REFACTOR_SCOPE`. A viability assessment (`/drupilot-assess`) always runs first as a decision gate and still emits a staged plan even when effort is high. **Preservation gate:** the adapted test suite staying green is the evidence behavior is preserved (both phases). Test adaptations may change only a test's *form* (PHPUnit/Drupal API), never *what it verifies* — a behavioral failure is a production-code regression fixed in code, never a test relaxed to fake green. `run-phpunit.sh` persists the honest verdict to `last-test.json` as `preservation`: `verified` / `verified-partial` (some groups skipped) / `regression` / `not-verified-blocked` / `not-verified-no-tests` (drupilot does not fabricate tests in Phase 1). A per-port **report card** (`port-report.sh` → `port-report.md`, from a `port-manifest.json` the flow writes) records what changed and why.

- **Run modes of the `/drupilot` router** (an explicit mode word wins; otherwise the mode is **inferred from intent** — a natural-language port request like "port this to D11" runs the flow as `full`, a bare `/drupilot` or "what's next" just recommends): `next` summarizes + recommends one step (the exploratory default); `status` is read-only; `full` runs the whole pipeline **with** confirmations and leaves refactor/contribute opt-in; `auto` (≡ `DRUPILOT_AUTONOMOUS=true`) runs `setup→assess→port→refactor→test` **hands-off** — no initial confirmation, `DRUPILOT_GENERATE_RULES` treated as `auto`, and it **never** does anything outward-facing (no push/MR/contribute, even in `auto` contribution mode). `auto` only relaxes drupilot's own gates; the Claude Code permission mode still governs Bash/Edit/Write.

- **Decision points & interaction model:** the developer owns the consequential forks, surfaced as **tabbed choices** — `AskUserQuestion` in commands (it is in their `allowed-tools`), with `choose_one()` as the fail-safe shell fallback. The recommendation is the pre-selected default; the answer persists to `.drupilot.json` via `prefs_set`. Canonical forks: router intent (full/next/auto), the **core target** (`/drupilot-port`, from `core-strategy.sh --json`), the **PHP target** (`/drupilot-setup`), the **digests rules to apply** (participatory per-rule review, pre-flagging rules whose target API exceeds the kept floor), the **refactor scope** (`/drupilot-refactor` multi-select → `DRUPILOT_REFACTOR_SCOPE` + PHPStan level), the **end-of-stage "what next?"**, and contribute **mode/push**. Outward-facing options are conditioned on `autonomous=false`; an autonomous run presents no interactive tabs and resolves with safe defaults. The next-step ladder (`doctor→setup→assess→port→[refactor]→test→[contribute]`) lives in one place, `scripts/env/next-step.sh` (router + `/drupilot-status` call it; never restate it in prose).

- **Two patch kinds, one script (`scripts/contrib/make-patch.sh`) — patch is decoupled from contribution:** `--local` writes an offline preview patch next to the module (git-only, no rebase, no `contribute` gate; new files captured via a throwaway `GIT_INDEX_FILE` so the real index is untouched) — emitted automatically at the end of port/refactor, and on demand via the first-class **`/drupilot-patch`** command (model-invocable, no gate). A bare `--local` is named `MODULE-port-to-drupal-11.patch`; `--local --issue ID [--comment N]` names it with the Drupal.org **issue-comment** convention (`MODULE-…-ISSUE-COMMENT.patch`) **while staying offline** — to attach to an issue and test now, contributing the MR later. The legacy/contribution mode rebases onto `origin/BASE` and **verifies the patch applies cleanly onto that base** (a **hard gate** — a non-applying patch is discarded and the script exits non-zero; the `--local` preview only warns), produced **always alongside the MR**. A companion `scripts/contrib/make-issue.sh` generates the paste-ready Drupal.org issue summary + recommended mandatory-field values (`DRUPILOT_ISSUE_*`) and the MR/issue comment (phase-aware via `--phase port|refactor`); `open-mr.sh --description-file` uses that comment as the MR description.

- **PHP floor vs. target compatibility (`scripts/analysis/detect-php-floor.sh` + `recommend_core_target`):** a heuristic scan classifies the PHP a module's code requires. It feeds two things: the composer `require.php` floor when keeping `^10 || ^11` (`DRUPILOT_REQUIRE_PHP_FLOOR`, default `detect`, widens it to the real floor, e.g. `>=8.1`, vs `target` which keeps `>=<target>`), and a `php_floor_target_compatible` verdict (false when the code uses a construct newer than the target). It is a cheap static signal, **not** authoritative — the test suite running on the target PHP inside DDEV is the real proof. A kept `^10 || ^11` is reported `declared-not-verified` (mirrors the preservation gate); `make-issue.sh --d10-unverified` carries the "verify Drupal 10 compatibility" task into the issue.

- **The `dbuytaert/drupal-digests` layer** is third-party, AI-generated, and **unlicensed**: it is cloned into a runtime cache (`digests_cache_dir`, under `${CLAUDE_PLUGIN_DATA}`) and referenced by path — **never vendored or committed**. It runs after official `drupal-rector`, always dry-run → review → apply → validate, and is filtered by the target `core_version_requirement` so it does not silently raise the minimum to 11.2+. Toggle with `DRUPILOT_USE_DIGESTS_RULES`. Its default ref stays `main`, but in deterministic mode `run-rector.sh` resolves `main` to a commit SHA, **freezes it in the lockfile and verifies the checkout** (reclones rather than reusing a stale cache), so the AI rules stay fixed per project.

- **Insight tools (drupal.org-aware, degrade gracefully):** `scripts/analysis/deps-status.sh` classifies each contrib dependency's D11 readiness against the drupal.org release-history feed (`ready`/`not-ready`/`not-on-drupalorg`/`unknown` — never guessed); `scripts/contrib/find-upstream-issue.sh` checks the issue queue for an existing D11 effort (best-effort title scan via the api-d7 feed, always with the filtered issue-queue URL as the reliable fallback); `scripts/analysis/explain-deprecations.sh` annotates analyzer output with what changed + the fix + a drupal.org change-records link, driven by the curated `config/deprecations.json` map (patterns are POSIX **ERE** — `grep -E` — so no PCRE lookarounds; the change-record link is a deterministic search URL keyed by the symbol, never a hardcoded node id). All three offer `--json` and never block a port on a blocked network.

- **Plugin layout rule:** only `plugin.json` belongs in `.claude-plugin/`; all component directories (`commands/`, `agents/`, `skills/`, `hooks/`, `scripts/`, `templates/`) live at the plugin root. `marketplace.json` also sits in `.claude-plugin/` and resolves plugin `source` (`"./"`) relative to the repo root. Hook/command paths use the `"${CLAUDE_PLUGIN_ROOT}"`-prefixed form.

## Conventions when adding or editing

- **New shell script:** `#!/usr/bin/env bash`, then `set -euo pipefail` (hook scripts excepted — see above), source `common.sh` at the correct `../` depth, parse args with a `while/case` loop, gate heavy/destructive operations through `preflight.sh`, keep it idempotent (detect-and-skip), English messages only, and `chmod +x` it. Reuse the `common.sh` helpers (logging, config, prefs, `choose_one`, lock) rather than reinventing them; keep stdout parseable and add a `--json` mode when a command needs to read counts/data. Watch the `set -e` traps: a function or pipeline whose last command returns non-zero (e.g. a `[[ ]] &&` tail, `grep -c` with no match) aborts the script — end such helpers with `return 0`. Reference the script's documented CLI from any command/skill that calls it.
- **New command:** `commands/<name>.md` with YAML frontmatter (`description`, `argument-hint`, `allowed-tools`; add `disable-model-invocation: true` for outward-facing/destructive commands like `drupilot-contribute`). Body is a prompt template that gates requirements, loads the relevant skill, drives scripts, and summarizes in English.
- **New decision point:** surface a consequential fork as a tab — add `AskUserQuestion` to the command's `allowed-tools`, make the recommendation the default option, persist the answer with `prefs_set` (`.drupilot.json`), gate outward-facing options on `autonomous=false`, and mirror the choice in the relevant skill/agent. The shell-side counterpart is `choose_one()`.
- **New skill/subagent:** `skills/<dir>/SKILL.md` / `agents/<name>.md` — embed the relevant versions/commands from `../PROMPT.md` §1–§3 so they never re-research.
- **Templates** (`templates/*.tmpl`) use `{{PLACEHOLDER}}` tokens substituted at setup time and are parameterized by the PHP target.

## Changelog and releases

Record **every notable change in `CHANGELOG.md`** under the `[Unreleased]` section as you make it (Keep a Changelog format: `Added` / `Changed` / `Deprecated` / `Removed` / `Fixed` / `Security`). This is the running log of improvements — do not skip it.

On a release, keep the version in sync across three places: rename `[Unreleased]` to the new version with a date in `CHANGELOG.md`, bump `version` in `.claude-plugin/plugin.json` **and** the `drupilot` entry in `.claude-plugin/marketplace.json` to match, then tag the commit `vX.Y.Z` (the `CHANGELOG.md` compare/tag links assume those tags exist). Follow Semantic Versioning.
